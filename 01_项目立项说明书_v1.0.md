## 2. 项目背景

维修行业长期存在「配件不透明、小病大修、以次充好」的顽疾。啄木鸟、58 等平台虽聚合了维修师傅，但并未从根本上解决信任问题。

**「维修透明工具平台」的定位是**：
> 不碰交易、不抽成、不派单、不烧融资的**轻量维修工具平台**，核心解决两个刚需：
> 1. **师傅**：多接单、好成交、留客户
> 2. **消费者**：配件透明、价格可查、不怕被坑

**旧件回收闭环**是该平台最核心的信任机制之一：通过**强制回收更换下来的旧配件**，从源头压缩「偷换配件」「小病大修」的操作空间，同时将旧件分类处置形成额外收益。

本项目即为将该闭环机制落地为**可运行的数字化系统**。

## 3. 项目目标

### 3.1 总体目标

建立废旧配件回收数字化管理系统，实现维修旧件从**回收到处置的全流程追踪**，为维修平台建立可信的旧件闭环能力。

### 3.2 分项目标（可量化）

| 编号 | 目标项 | 量化指标 | 验收标准 |
|:---:|:---|:---|:---|
| G1 | 旧件回收线上化 | 100% 的更换配件订单需上传旧件凭证 | 系统强制校验，未上传则无法标记工单完成 |
| G2 | 回收流程可追溯 | 单件旧件从回收到处置全链路可查 | 任意旧件 30 秒内可查来源工单、回收时间、处置方式 |
| G3 | 师傅激励可量化 | 回收积分自动发放，兑换规则透明 | 回收确认后 24 小时内积分到账，积分规则后台可配置 |
| G4 | 管理后台可运营 | 支持入库审核、分类处置、数据统计 | 后台审核响应时间 < 24h，分类处置标记完整 |
| G5 | 系统可扩展 | 预留 AI 图像识别接口、预留对接回收商 API | 接口文档完备，后续接入无需改动核心表结构 |

## 4. 业务痛点

### 4.1 行业痛点（现状）

| 痛点 | 现象描述 | 后果 |
|:---|:---|:---|
| **偷换配件** | 师傅用低价劣质件替换原配件，谎称已更换核心部件 | 用户多花钱、设备寿命缩短 |
| **小病大修** | 小故障被虚报为核心配件损坏，骗取高额维修费 | 用户信任崩塌、行业口碑恶化 |
| **旧件去向不明** | 更换下来的旧件被师傅自行带走处理，用户无从核实 | 无法验证「是否真实维修」 |
| **缺乏激励** | 师傅回收旧件无正向回报，执行意愿低 | 规则形同虚设，无法落地 |

### 4.2 现有解决方案不足

- **口头约定**：师傅承诺回收，但无凭证、无追溯，全靠自律
- **纸质登记**：效率低、易丢失、无法跨地区协同
- **第三方回收平台**：与维修流程脱节，无法形成「维修→回收」闭环

## 5. 目标用户与应用场景

### 5.1 用户角色定义

| 角色 | 职责 | 核心诉求 |
|:---|:---|:---|
| **维修师傅**（B端核心用户） | 上门维修、更换配件、拍摄新旧件对比照片、寄送旧件 | 操作简单、积分奖励及时、不额外增加负担 |
| **平台管理员**（运营方） | 审核旧件入库、分类处置、发放积分奖励、处理异常 | 审核流程清晰、处置路径标准化、数据可统计 |
| **回收商/合作方**（外部） | 接收可翻新旧件、提供环保合规的电子垃圾处理 | 货源稳定、品类清晰、结算透明 |

### 5.2 关键应用场景

#### 场景一：维修中拍照留证（师傅端）

```
触发：师傅完成配件更换
动作：
  1. 拍摄「新配件未安装时的清晰照片（带型号标识）」
  2. 拍摄「旧配件拆下后的清晰照片（带设备型号标识）」
  3. 上传至平台工单系统
约束：未上传旧件照片，系统禁止标记工单「已完成」
```

#### 场景二：旧件回收入库（师傅端 + 管理后台）

```
触发：师傅积累一定数量的旧件后
动作：
  1. 师傅在系统中发起「回收寄送」申请
  2. 按平台分类标准打包旧件，寄送至指定回收点
  3. 回收点（管理员）在系统中确认收货、核对数量与类型
  4. 系统自动按规则发放积分奖励到师傅账户
```

#### 场景三：旧件分类处置（管理后台）

```
触发：回收点收到旧件后
动作：
  1. 管理员在后台进行入库审核，核对与师傅申报是否一致
  2. 标记处置分类：
     ├─ 可翻新/拆解 → 联系二手维修商/配件回收商估价
     ├─ 不可回收电子垃圾 → 移交有资质环保机构，留存处理凭证
     └─ 异常件 → 标记待核查
  3. 处置结果录入系统，闭环完成
```

#### 场景四：用户自留备案（异常流程）

```
触发：用户要求自留旧件
动作：
  1. 师傅在工单中提交「用户自留」申请
  2. 上传旧件照片备案
  3. 系统自动将该工单标记为「高风险关注单」
  4. 平台后续重点核查该工单评价与投诉
```

## 6. 商业价值分析

### 6.1 直接收益

| 收益来源 | 说明 | 预估模式 |
|:---|:---|:---|
| **可翻新配件二次销售** | 功能完好的旧件经检测后，可转售给二手维修市场或配件回收商 | 按品类定价，平台抽成或赚差价 |
| **环保处理服务费** | 不可回收电子垃圾移交有资质机构，可收取合规处理服务差价 | 按重量/品类计费 |
| **积分兑换权益** | 师傅积分可兑换平台高级会员权益，形成会员收入 | 非现金、合规积分体系 |

### 6.2 间接价值（核心壁垒）

| 价值维度 | 说明 |
|:---|:---|
| **信任壁垒** | 行业内首个实现「旧件回收全流程可追溯」的轻量平台，形成差异化竞争 |
| **师傅粘性** | 回收积分激励提高师傅留存，降低被其他平台挖走的风险 |
| **合规背书** | 电子垃圾合规处理凭证可作为平台社会责任背书，便于后续政府合作 |
| **数据资产** | 积累真实的「故障→配件→旧件」数据集，为后续 AI 模型训练提供标注数据 |

### 6.3 成本估算

| 成本项 | 说明 | 预估（第一期） |
|:---|:---|---:|
| 系统开发 | 2 后端 + 1 前端 + 1 产品，6 周 | 人力成本为主 |
| 服务器/存储 | 轻量云服务器 + 对象存储（图片） | < 500 元/月 |
| 回收点合作 | 与本地 1-2 家回收商合作，无需自建仓库 | 按件结算，前期无固定成本 |

## 7. 系统边界与范围（明确不做）

以下范围**本期明确不做**，需在文档和需求评审中严格把控：

| 编号 | 不做事项 | 原因 | 后续规划 |

### 8.2 旧件状态流转

| 状态 | 说明 | 触发条件 | 操作人 |
|:---|:---|:---|:---|
| `已拆下` | 旧件已从设备拆下，照片已上传 | 师傅上传旧件照片 | 维修师傅 |
| `待寄送` | 师傅已确认回收，等待打包寄送 | 师傅在系统中选择「平台回收」 | 维修师傅 |
| `运输中` | 旧件已发出，物流途中 | 师傅填写物流单号 | 维修师傅 |
| `待入库` | 回收点已收到，等待管理员核对 | 物流签收 | 系统自动 |
| `已入库` | 管理员核对无误，确认入库 | 管理员点击「确认入库」 | 平台管理员 |
| `已分类` | 已完成分类处置标记 | 管理员选择处置分类 | 平台管理员 |
| `已处置` | 旧件已完成最终处理（卖出/环保处理） | 处置完成凭证上传 | 平台管理员 |
| `异常` | 与申报不符或损坏无法回收 | 管理员标记异常 | 平台管理员 |

## 9. 技术架构建议

### 9.1 架构选型结论

| 层级 | 选型 | 说明 |
|:---|:---|:---|
| **前端** | 轻量 Web 管理后台（Vue3 / React） | 管理员日常操作，无需小程序 |
| **API** | RESTful API（FastAPI / Express） | 标准化接口，供母项目小程序调用 |
| **数据库** | SQLite（快速启动）→ 后续迁移 MySQL | 第一期数据量可控，SQLite 零运维；预留迁移方案 |
| **文件存储** | 本地存储 / 云对象存储（可选） | 旧件照片存储，支持按需升级 |
| **部署** | 单服务器 Docker 容器化 | 轻量、可迁移、易回滚 |

### 9.2 架构优势

1. **启动快**：SQLite 无需独立数据库服务器，一行命令启动
2. **成本低**：单台轻量云服务器即可承载，适合 MVP 验证
3. **扩展顺**：预留标准数据库迁移脚本，业务量增长后平滑切换 MySQL
4. **接口清**：RESTful API 与母项目小程序完全解耦，独立迭代

### 9.3 核心模块划分

```
废旧回收管理程序
├── 回收工单模块          # 与母项目订单关联，管理旧件回收生命周期
├── 拍照凭证模块          # 新旧件照片上传、存储、校验
├── 回收入库模块          # 寄送申请、物流跟踪、入库确认
├── 积分激励模块          # 积分规则配置、自动发放、积分流水
├── 分类处置模块          # 处置分类标记、合作方对接（预留）
├── 数据统计模块          # 回收量、处置量、积分发放统计
└── 系统管理模块          # 管理员账户、权限、配置参数
```

| 风险项 | 概率 | 影响 | 应对策略 |
|:---|:---:|:---:|:---|
| 师傅使用意愿低 | 中 | 高 | 回收积分与师傅等级/接单权限挂钩；简化拍照流程至最少步骤 |
| 旧件物流成本高 | 中 | 中 | 设定最低回收数量门槛（如满 5 件或满 1kg 才寄送）；与本地回收商合作降低距离 |
| 旧件分类标准不统一 | 低 | 中 | 上线前制定详细分类手册（含图文示例）；预留分类规则后台可配置 |
| 积分规则引发争议 | 中 | 中 | 规则后台透明可配置，变动提前公示；第一期采用固定积分制，不引入浮动定价 |
| 数据量增长超 SQLite 承载 | 低 | 高 | 预留 MySQL 迁移脚本；设置数据量监控告警（> 10 万条触发评估） |
| 合规风险（环保处理） | 低 | 高 | 仅与持有「废弃电器电子产品处理资质」的合作方签约；留存处理凭证 |

## 12. 关键结论

| 决策项 | 结论 | 理由 |
|:---|:---|:---|
| **系统形态** | 轻量 Web 管理后台 + RESTful API | 管理员操作以 Web 为主；API 供母项目小程序调用，解耦迭代 |
| **数据库** | SQLite 快速启动，预留 MySQL 迁移 | 第一期数据量小、零运维成本低；标准迁移脚本保障平滑过渡 |
| **AI 识别** | 本期不做，预留接口 | 降低 MVP 复杂度；先用人工审核 + 强制拍照保障基础闭环 |
| **资金处理** | 纯积分激励，不碰现金 | 规避二清、资金池等合规风险；积分与平台权益挂钩 |
| **部署方式** | 单服务器 Docker 容器化 | 轻量、低成本、可回滚、易迁移 |
| **与母项目关系** | 独立子系统，API 集成 | 可独立开发测试；母项目小程序通过 API 调用本系统能力 |
