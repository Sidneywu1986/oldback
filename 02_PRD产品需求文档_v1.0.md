## 1. 文档概述

### 1.1 项目背景

维修透明工具平台以「配件透明、价格可查、不怕被坑」为核心定位，旧件回收闭环是防止「小病大修、偷换配件」的关键抓手。本PRD聚焦**废旧回收管理程序**，覆盖旧件从师傅上传、平台审核、入库处置到积分发放的全生命周期管理。

### 1.2 核心目标

| 目标编号 | 目标描述 | 衡量指标 |
|---------|---------|---------|
| G1 | 将旧件回收嵌入维修流程，成为工单完成的必要条件 | 旧件上传率 ≥ 95% |
| G2 | 建立可追溯的旧件回收闭环，压缩欺诈空间 | 纠纷率下降 ≥ 30% |
| G3 | 通过积分激励提升师傅执行积极性 | 师傅回收参与率 ≥ 80% |
| G4 | 实现旧件分类处置，产生额外收益 | 月处置旧件量 ≥ 500件 |

### 1.3 术语表

| 术语 | 定义 |
|-----|------|
| 旧件 | 维修过程中被更换下来的废旧配件 |
| 师傅端 | 维修师傅使用的微信小程序端 |
| 管理后台 | 平台运营人员使用的Web管理端 |
| 工单 | 与用户维修订单绑定的服务单据 |
| 自留备案 | 用户申请保留旧件时的线上备案流程 |
| 高风险单（L3） | 标记为需重点核查的异常订单 |
| 积分 | 师傅回收旧件后获得的平台奖励点数，可兑换权益 |
| 处置 | 平台对入库旧件进行分类处理（翻新/回收/环保处理） |

### 1.4 系统边界

```
┌─────────────────────────────────────────────────────────────┐
│                    废旧回收管理程序 系统边界                   │
├─────────────────────────────────────────────────────────────┤
│  包含：                                                       │
│    ✓ 师傅端旧件上传、自留申请、回收记录查看                    │
│    ✓ 管理后台数据看板、旧件审核、入库管理、奖励配置            │
│    ✓ 回收品类配置、处置方式配置、师傅信用分查看                │

│    ✗ AI故障识别与核价（归属AI服务模块）                        │
│    ✗ 纠纷仲裁与处罚（归属纠纷模块）                            │
└─────────────────────────────────────────────────────────────┘
```

## 3. 用户操作流程

### 3.1 主流程一：师傅回收旧件闭环流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    师傅回收旧件 → 拍照上传 → 平台审核 → 确认入库 → 发放积分          │
└─────────────────────────────────────────────────────────────────────────────┘

  [师傅端小程序]                              [管理后台Web]                      [系统自动化]
       │                                           │                              │
       │  ① 维修完成，更换配件                      │                              │
       │─────────────────────────────────────────>│                              │
       │                                           │                              │
       │  ② 拍摄照片（新件未装前+旧件拆下后）        │                              │
       │  ├─ 照片①：新配件未安装时清晰照片（带型号）  │                              │
       │  └─ 照片②：旧配件拆下后清晰照片（带设备标识）│                              │
       │                                           │                              │
       │  ③ 上传旧件照片到工单                      │                              │
       │  ├─ 选择关联配件名称                       │                              │
       │  ├─ 选择「平台回收」                       │                              │
       │  └─ 确认提交                               │                              │
       │─────────────────────────────────────────>│                              │
       │                                           │                              │
       │                              ④ 系统校验照片完整性                      │
       │                              ├─ 是否上传≥2张照片？                      │
       │                              ├─ 照片清晰度是否达标？                     │
       │                              └─ 是否关联有效工单？                       │
       │                                           │                              │
       │                              ⑤ 照片校验通过 → 状态：待审核               │
       │                              ⑥ 照片校验不通过 → 退回重传               │
       │<─────────────────────────────────────────│                              │
       │                                           │                              │
       │                              ⑦ 审核员登录后台查看待审核列表              │
       │                                           │                              │
       │                              ⑧ 审核员核验旧件与工单匹配性               │
       │                              ├─ 配件名称与工单一致？                     │
       │                              ├─ 照片能清晰识别旧件？                     │
       │                              └─ 是否存在欺诈嫌疑？                       │
       │                                           │                              │
       │                              ⑨ [审核通过] → 状态：已确认                 │
       │                              ⑩ [审核驳回] → 状态：待重传+备注原因         │
       │<─────────────────────────────────────────│                              │
       │                                           │                              │
       │ ⑪ 师傅收到驳回通知，重新拍照上传           │                              │
       │─────────────────────────────────────────>│  → 回到步骤⑤                │
       │                                           │                              │
       │                              ⑫ 仓库收货 → 状态：已入库                   │
       │                              ⑬ 标记处置方式 → 状态：已处置               │
       │                                           │                              │
       │                              ⑭ 系统自动计算积分并发放                    │
       │                              ├─ 根据品类×重量匹配积分规则                 │
       │                              └─ 写入师傅积分账户                         │
       │<─────────────────────────────────────────│                              │
       │                                           │                              │
       │ ⑮ 师傅收到积分到账通知                     │                              │
       │ ⑯ 师傅可在「积分明细」查看记录             │                              │
       │                                           │                              │
       ▼                                           ▼                              ▼

                              【流程完结：闭环达成】
```

## 5. 异常场景与处理机制

### 5.1 异常场景总览

| 异常编号 | 异常场景 | 触发条件 | 影响范围 | 处理优先级 |
|---------|---------|---------|---------|----------|
| E1 | 照片不清晰 | 上传照片模糊/过暗/无法识别 | 当前回收单 | P0 |
| E2 | 旧件与工单不匹配 | 照片显示配件与工单申报不一致 | 当前回收单+工单 | P0 |
| E3 | 用户申请自留 | 用户要求保留旧件 | 当前工单 | P0 |
| E4 | 批量回收 | 同一工单更换多个配件 | 当前回收单 | P1 |
| E5 | 旧件损坏无法回收 | 旧件在拆卸过程中损坏 | 当前回收单 | P1 |
| E6 | 师傅逾期未寄回 | 已确认但超过7天未寄到仓库 | 当前回收单 | P1 |
| E7 | 积分发放失败 | 系统异常导致积分未到账 | 当前回收单 | P0 |
| E8 | 重复提交 | 同一工单多次提交旧件 | 当前回收单 | P1 |

---

### 5.2 异常场景详细说明

#### E1：照片不清晰 → 退回重传

```
触发条件：
  └─ 照片分辨率 < 1280×720
  └─ 照片文件大小 < 100KB（疑似过度压缩）
  └─ AI识别无法提取有效配件轮廓
  └─ 人工审核判断照片无法清晰识别旧件

处理流程：
  ① 系统/人工标记「照片不合格」
  ② 状态变更为「待重传」
  ③ 推送通知给师傅：「旧件照片不清晰，请重新拍摄上传。原因：{具体原因}」
  ④ 师傅收到通知后进入工单 → 重新拍摄 → 重新提交
  ⑤ 重新提交后回到「待审核」状态
  ⑥ 同一单最多允许重传3次，超过则自动转为「人工介入」状态

系统校验规则：
  ├─ 自动校验：分辨率、文件大小、清晰度算法评分
  ├─ 人工校验：审核员放大查看后判断
  └─ 强制要求：2张照片（新件未装前 + 旧件拆下后）

通知模板：
  「【退回重传】您的旧件回收单 {回收单号} 照片审核未通过，
   原因：{驳回原因}。请重新拍摄上传，剩余重传次数：{N}次。」
```

#### E2：旧件与工单不匹配 → 驳回并备注

```
触发条件：
  └─ 旧件照片显示的配件与工单申报的配件名称不一致
  └─ 旧件照片显示的设备型号与工单设备型号不一致
  └─ 照片中有明显PS/拼接痕迹（人工判断）
  └─ 多张旧件照片存在时间/光照/环境明显不一致（疑似套图）

处理流程：
  ① 审核员在后台标记「旧件与工单不匹配」
  ② 状态变更为「待重传」或「已关闭」（视严重程度）
  ③ 填写详细备注说明不匹配点
  ④ 推送通知给师傅，同时抄送运营主管
  ⑤ 如属欺诈嫌疑：触发信用分扣减，工单标记L3高风险
  ⑥ 连续2次不匹配：师傅进入重点观察名单

信用分影响：
  ├─ 首次不匹配：不扣分，仅教育通知
  ├─ 第二次不匹配：扣减信用分5分
  ├─ 第三次及以上：扣减信用分10分 + 人工复核
  └─ 确认欺诈：扣减信用分50分 + 暂停接单权限

通知模板：
  「【审核驳回】您的旧件回收单 {回收单号} 经核验与工单信息不匹配，
   原因：{具体原因}。请核对后重新提交，如有疑问请联系平台客服。」
```

#### E3：用户申请自留 → 触发高风险标记流程

```
触发条件：
  └─ 用户在维修现场要求保留旧件
  └─ 师傅在小程序中选择「用户自留」选项
  └─ 用户点击确认「知情同意自留」

处理流程：
  ① 师傅选择「用户自留」→ 系统弹出「旧件回收规则告知书」
  ② 用户阅读并点击「确认自留」→ 系统记录用户同意时间戳
  ③ 师傅必须上传旧件备案照片（≥2张）
  ④ 系统将该工单自动标记为 L3 高风险单
  ⑤ 工单状态旁显示红色「高风险」标签
  ⑥ 后台「自留备案」列表自动新增一条记录
  ⑦ 运营人员定期（每周）抽查自留备案单
  ⑧ 抽查方式：电话回访用户确认旧件是否仍保留
  ⑨ 如用户反馈旧件已丢失/被师傅带走 → 升级纠纷处理
  ⑩ 工单完结后，高风险标记保留90天，期间该用户/师傅的新工单自动提升关注等级

高风险标记规则：
  ├─ 单次自留：标记L3，重点观察
  ├─ 同一师傅30天内2次自留：标记L2，人工必审
  ├─ 同一师傅30天内3次及以上自留：标记L1，暂停接单调查
  └─ 同一用户多次自留不同师傅：标记用户侧高风险，回访频次提升

备案照片要求：
  ├─ 照片①：旧件全景照（能识别整体外观）
  ├─ 照片②：旧件细节照（能识别型号/序列号）
  └─ 照片③（可选）：用户手持旧件与师傅同框确认照

通知模板（师傅）：
  「【备案完成】该工单已标记为用户自留，已生成备案记录。
   工单已升级为高风险单，请确保服务规范，平台将进行回访核查。」

通知模板（运营）：
  「【高风险提醒】工单 {工单号} 触发用户自留备案，
   师傅：{师傅姓名}，已自动标记L3高风险，请纳入本周抽查名单。」
```

#### E4：批量回收 → 支持一单多配件

```
触发条件：
  └─ 同一维修工单更换了多个配件（如空调维修同时更换压缩机和电容）

处理流程（MVP简化版）：
  ① 师傅进入旧件上传页，系统自动识别工单中多个已更换配件
  ② 师傅逐个选择配件 → 逐个上传对应旧件照片
  ③ 每个配件生成独立的回收子单，共享同一母单号
  ④ 所有子单审核通过后方可触发工单完结

处理流程（二期完整版）：
  ① 支持「批量模式」一次上传多张旧件照片
  ② AI自动识别照片对应配件（辅助匹配）
  ③ 师傅确认AI匹配结果，可手动修正
  ④ 一键提交所有旧件，生成批量回收单
  ⑤ 审核时可批量通过/批量驳回
  ⑥ 批量入库、批量发放积分

数据模型：
  ├─ 母单：recycle_batch（批量回收单）
  │   └─ 关联order_id、master_id、总配件数
  └─ 子单：recycle_item（单件回收明细）
      └─ 关联batch_id、配件名称、照片、独立状态、独立积分

界面设计（二期）：
  ├─ 页面顶部：工单信息 + 配件清单卡片
  ├─ 每个配件独立卡片：照片上传区 + 状态指示
  ├─ 底部：总进度条（如 2/3 已完成上传）
  └─ 提交按钮：「全部提交」/「暂存草稿」
```

#### E5：旧件损坏无法回收 → 豁免申请

```
触发条件：
  └─ 旧件在拆卸过程中损坏（如锈蚀严重碎裂、电路板烧毁）
  └─ 旧件属于不可回收品类（如一次性耗材）
  └─ 旧件体积过小无法独立回收（如密封圈、垫片）

处理流程：
  ① 师傅在旧件上传页选择「损坏豁免」
  ② 填写豁免原因说明（必填，至少10字）
  ③ 上传损坏旧件照片（证明确实损坏/不可回收）
  ④ 提交后进入「豁免审核」状态
  ⑤ 审核员核实照片与说明是否一致
  ⑥ 审核通过 → 状态「已关闭（无需回收）」→ 工单可正常完结
  ⑦ 审核不通过 → 退回重传或要求强制回收

豁免规则：
  ├─ 每个师傅每月豁免次数 ≤ 3次（超出需说明）
  ├─ 同一工单豁免配件数 ≤ 总配件数的50%
  ├─ 高价值配件（如压缩机、主板）原则上不允许豁免
  └─ 连续豁免触发运营关注

通知模板：
  「【豁免申请】师傅 {师傅姓名} 申请工单 {工单号} 的旧件豁免回收，
   原因：{豁免说明}。请审核员及时处理。」
```

#### E6：师傅逾期未寄回 → 提醒与处罚

```
触发条件：
  └─ 旧件状态「已确认」后超过7个自然日
  └─ 仓库仍未收到对应旧件且未标记入库

处理流程：
  ① 第5天：系统自动推送提醒通知给师傅
  ② 第7天：系统再次推送警告通知
  ③ 第10天：自动标记「逾期」，信用分扣减3分
  ④ 第15天：运营人员电话跟进，确认旧件去向
  ⑤ 第30天：视为违约，信用分扣减10分，暂停新工单派发
  ⑥ 师傅补寄旧件后，信用分不恢复但停止继续扣分

提醒模板：
  「【逾期提醒】您的旧件回收单 {单号} 已确认通过，
   请尽快将旧件寄往平台指定回收点。已逾期{N}天，
   逾期将影响您的信用分和接单权限。」
```

#### E7：积分发放失败 → 自动重试+人工补偿

```
触发条件：
  └─ 系统执行积分发放时数据库异常
  └─ 师傅账户状态异常（如冻结）
  └─ 并发冲突导致积分写入失败

处理流程：
  ① 系统检测到发放失败，记录失败日志
  ② 自动重试机制：间隔5分钟重试，最多3次
  ③ 3次均失败：标记「发放失败」，写入异常队列
  ④ 运营人员每日查看异常队列
  ⑤ 人工核实后手动补发或联系师傅核实账户状态
  ⑥ 补发成功后更新发放记录状态

补偿规则：
  ├─ 系统原因导致延迟发放：补发+补偿10%积分
  ├─ 师傅账户问题：通知师傅修正后补发
  └─ 积分规则变更导致歧义：按有利于师傅的方式执行
```

#### E8：重复提交 → 去重拦截

```
触发条件：
  └─ 同一工单同一配件多次点击提交
  └─ 网络延迟导致重复请求

处理机制：
  ① 前端防抖：提交按钮点击后锁定3秒
  ② 后端幂等：以「工单ID+配件名称」为唯一键
  ③ 重复请求返回「该配件已提交，请勿重复上传」
  ④ 如确需补充照片，走「修改补充」流程而非重新提交

修改补充流程：
  ├─ 状态为「待审核」：允许撤回重新上传
  ├─ 状态为「待重传」：直接覆盖重传
  └─ 其他状态：不允许修改，需联系运营
```

| AC-10.4 | 并发审核 | 10个审核员同时操作，无数据冲突、状态错乱 | 并发压力测试 |
| AC-10.5 | 导出性能 | 导出10000条记录Excel ≤ 10秒 | 构造大数据量测试 |

### 6.6 安全与合规验收标准

| 验收编号 | 验收项 | 验收标准 | 验证方法 |
|---------|--------|---------|---------|
| AC-11.1 | 权限隔离 | 运营人员A无法操作需管理员权限的功能（如规则配置） | 越权访问测试 |
| AC-11.2 | 数据脱敏 | 师傅手机号在列表和详情中均脱敏展示（138****8888） | 前端检查+接口响应检查 |
| AC-11.3 | 操作日志 | 所有审核/入库/处置/积分发放操作均记录操作日志（操作人、时间、IP、内容） | 操作后查看日志表 |
| AC-11.4 | 照片防盗链 | 旧件照片URL带时效签名，过期后无法访问，非授权域名无法嵌入 | URL过期测试+跨域测试 |
| AC-11.5 | SQL注入防护 | 所有筛选/搜索参数经预编译处理，无法注入恶意SQL | 安全渗透测试 |

### 7.1 核心状态枚举定义

| 状态码 | 状态名称 | 说明 | 可操作 |

| 2 | 损坏豁免 | 旧件损坏无法回收，需审核豁免 |

### 7.5 数据库核心表结构（recycle表）

```sql
CREATE TABLE `recycle` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `order_id` bigint NOT NULL COMMENT '关联工单ID',
  `master_id` bigint NOT NULL COMMENT '师傅ID',
  `parts_name` varchar(64) DEFAULT NULL COMMENT '配件名称',
  `img_url` varchar(255) DEFAULT NULL COMMENT '旧件照片URL（多图以逗号分隔）',
  `user_keep` tinyint DEFAULT 0 COMMENT '0-平台回收 1-用户自留 2-损坏豁免',
  `status` tinyint DEFAULT 0 COMMENT '0待审核 1已确认 2已入库 3已处置 4待重传 5已关闭 6积分已发放',
  `point` int DEFAULT 0 COMMENT '发放积分',
  `audit_remark` varchar(255) DEFAULT NULL COMMENT '审核备注',
  `audit_user` varchar(32) DEFAULT NULL COMMENT '审核人',
  `audit_time` datetime DEFAULT NULL COMMENT '审核时间',
  `warehouse_user` varchar(32) DEFAULT NULL COMMENT '入库人',
  `warehouse_time` datetime DEFAULT NULL COMMENT '入库时间',
  `dispose_type` varchar(32) DEFAULT NULL COMMENT '处置方式编码',
  `dispose_time` datetime DEFAULT NULL COMMENT '处置时间',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_order_id` (`order_id`),
  KEY `idx_master_id` (`master_id`),
  KEY `idx_status` (`status`),
  KEY `idx_create_time` (`create_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='旧件回收表';
```
